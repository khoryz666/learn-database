Oracle Overview

Database Systems
◼A Database System consists of:
1.a DBMS (Database Management System), and 
2.One or more Database Applications. (Ex: SQLPlus , Oracle 
Forms, and PowerBuilder)
◼There are two main types of database systems: 
1.Personal Databases (Ex: MS Access, where database 
application and the DBMS reside on the same machine and 
exist as a single process) and 
2.Client/Server Databases (Ex: Oracle21c is a client/server 
database, where many users can use client/sever databases 
simultaneously)
|-UCCD2303/UCCD2203 -| 4

Client/Server Database Systems
◼Client/Server databases store their data on a server 
machine that is usually backed up on a regular basis.
◼If a client machine fails, the server is still accessible 
to other client machines. 
◼A Client/Server database is appropriate for a high -
volume, multi -user, mission -critical application.

Database/User Privilege

|-UCCD2303/UCCD2203 -| 33User Privileges
The main types of  user privileges are as follows:
◼System privilege
❑A system privilege gives a user the ability to perform a particular action 
within the database.
❑For example, the system privilege CREATE TABLE permits a user to 
create tables, and the system privilege CREATE USER permits a user to 
create database users.
◼Object privilege
❑An object privilege gives a user the ability to perform a particular action 
on a specific schema object. 
❑The privilege to select rows from the EMPLOYEES table or to delete 
rows from the DEPARTMENTS table are examples of  object privileges.

Using Database 
Constraints to Achieve 
Data Integrity

49Constraints
◼Rules that restrict the data values that you can enter into a field 
in a database table
◼Constraint prevent the deletion of a table if there are 
dependencies.
◼The following constraint types are valid:
NOT NULL
UNIQUE 
PRIMARY KEY
FOREIGN KEY
CHECK
|-UCCD2303/UCCD2203 -|
50Oracle Constraint Naming Convention
tablename _ fieldname _constraintID
◦You can name a constraint, or the Oracle server generates a 
name by using the SYS_C nformat.
◦Create a constraint at either of  the following times:
At the same time as the table is created
After the table has been created
|-UCCD2303/UCCD2203 -|
51Creating Constraints
◼At column level
❑References a single column
CREATE TABLE client 
(cl_no NUMBER(4) CONSTRAINT client_cl_no_pk PRIMARY KEY, 
cl_first VARCHAR2(20), 
cl_last VARCHAR2(20));
◼At table level
❑Are listed after all the table columns have been defined.
CREATE TABLE client 
(cl_no NUMBER(4) NOT NULL, 
cl_first VARCHAR2(20), 
cl_last VARCHAR2(20)
CONSTRAINT client_cl_no_pk PRIMARY KEY( cl_no ));
|-UCCD2303/UCCD2203 -|
52Basic Rules For Constraints
◼Constraints that refer to more than one column 
must be defined at the table level.
◼The NOT NULL constraint can be specified only 
at the column level.
◼UNIQUE, PRIMARY KEY, FOREIGN KEY 
and CHECK constraints can be defined at either 
the column or table level.
◼If the word CONSTRAINT is used, you must give 
the constraint a name.
|-UCCD2303/UCCD2203 -|
53Integrity Constraints
◼Define primary key fields
Loc_id Number(6) PRIMARY KEY;
Loc_id Number(6) CONSTRAINT location_loc_id_pk PRIMARY KEY; 
CONSTRAINT location_loc_id_pk PRIMARY KEY ( loc_id ));
◼Specify foreign keys and their corresponding table and column 
references
CONSTRAINT faculty_loc_id_fk REFERENCES location( loc_id ));
CONSTRAINT faculty_loc_id_fk FOREIGN KEY ( loc_id ) 
REFERENCES location( loc_id ));
◼Specify composite keys
CONSTRAINT cust_agent_id_pk PRIMARY KEY ( cust_id , agent_id ));
|-UCCD2303/UCCD2203 -|
54Value Constraints
◼Check conditions : field value must be a specific value or fall 
within a range of values
s_class CHAR(2) CONSTRAINT student_s_class_cc
CHECK (( s_class = 'FR') OR ( s_class = 'SO') OR ( s_class = 
'JR') OR ( s_class = 'SR'))
◼NOT NULL constraints : specify whether a field value can be 
NULL
job_id VARCHAR2(10) NOT NULL
◼Unique constraints : specify that a field must have a unique value 
for every table record
CONSTRAINT emp_email_uk UNIQUE(email));
|-UCCD2303/UCCD2203 -|
CHECK constraint
◼The following expression are not allowed:
❑References to CURRVAL, NEXTVAL, ROWNUM 
Pseudocolumns .
❑Calls to SYSDATE, USER, UID, USERENV function.
❑Queries that refer to other values in other rows.
Violating Constraints
❑When you have constraints in  place on columns, an error 
is returned if you try to violate the constraint rules.

Default Option
◼Specify a default value for a column during an insert
◼Literal values, SQL functions are legal values
◼Another column’s name or pseudocolumn are illegal 
values
◼The default data type must match the column data 
type.
Eg:… hire_date DATE DEFAULT SYSDATE, …

Managing Constraints
◼You can add, drop, enable, disable a constraint, but 
you cannot modify its structure.
◼You can add a NOT NULL constraint to an 
existing column by using the MODIFY clause of 
the ALTER TABLE statement.
◼You can define a NOT NULL constraint only if 
the table is empty or if the column has a value for 
every row.
