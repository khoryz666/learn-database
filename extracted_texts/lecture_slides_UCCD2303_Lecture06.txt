Categories of Failure
1.Statement failure: A single database operation (select, insert, 
update, or delete) fails.
2.User process failure: A single database session fails.
3.Network failure: Connectivity to the database is lost.
4.User error: A user successfully completes an operation, but the 
operation (dropping a table or entering bad data) is incorrect.
5.Instance failure: The database instance shuts down unexpectedly.
6.Media failure: A loss of any file that is needed for database 
operation (that is, the files have been deleted or the disk has failed).

Recovery techniques using deferred update
We use the log file to protect against system failures in the 
following ways:
◼When a transaction starts, write a transaction start record to the 
log.
◼When any write operation is performed, write a log record 
containing the after -image of the update.
◼When a transaction is about to commit:
❑write a Transaction Commit log record,
❑write all the log records for the transaction to disk, then
❑Use the log records to perform the actual updates to the database.
◼If a transaction aborts, write a transaction abort log record and do 
not perform the writes,
❑A transaction without a Transaction Abort record is also aborted.

Recovery techniques using immediate update
We use the log file to protect against system failures in the 
following ways:
◼When a transaction starts, write a transaction start record to the 
log.
◼When any write operation is performed, write a log record 
containing the necessary data to the log file.
◼When the log record is written, write the update to the database 
buffers.
◼The updates to the database itself are written when the buffers 
are next flushed to secondary storage.
◼When the transaction commits, write a transaction commit record to 
the log.
43
Immediate Update -Write -ahead log protocol
It is essential that log records are written before 
write to database. Write -ahead log protocol . 
◼Log record writes must preceded database 
writes.
◼If updates were made to the database first, and 
failure occurred before the log record was 
written, then RMAN (Recovery Manager) would 
have no way of undoing or redoing the 
operation.
